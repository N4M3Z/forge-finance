# EPO XML attachments — `<Prilohy>` and `<Kontrola>`

PDF attachments and the EPO checksum element. Applies across Czech tax-form XML schemas (DPPDP9 for DPPO, DPFDP7 for DPFO, etc.) — not specific to one schema.

## Where the XML lands

EPO ("Elektronická podání pro Finanční správu") is the MF portal for filling, saving, and submitting tax-form XML. Two entry points:

| Use                                                                | URL                                                |
| ------------------------------------------------------------------ | -------------------------------------------------- |
| Browse forms, fill from scratch, save XML                          | [adisspr.mfcr.cz/pmd/epo][epo]                     |
| Load an existing XML to validate, edit, sign, submit               | [adisspr.mfcr.cz/pmd/epo/formulare?nacteni=1][load]|
| DPPDP9 schema reference (XSD, attribute docs)                      | [adisspr.mfcr.cz DPPDP9][dppdp9]                   |

Typical flow when filing a self-prepared XML: open the load page, upload the file, EPO validates and renders the form, attach any embedded `<Prilohy>` or supply PDFs through the UI, sign via DS, submit.

[epo]: https://adisspr.mfcr.cz/pmd/epo
[load]: https://adisspr.mfcr.cz/pmd/epo/formulare?nacteni=1
[dppdp9]: https://adisspr.mfcr.cz/dpr/adis/idpr_pub/epo2_info/popis_struktury_detail.faces?zkratka=DPPDP9

## Embedded attachments via `<Prilohy>`

PDF attachments can be embedded as base64 inside the XML, producing a single self-contained submission file. This is the EPO-native pattern and avoids the two-step "upload XML, then attach PDFs in portal UI" workflow.

Structure (sibling of the schema root element, appears after `VetaUZ` for DPPDP9):

```xml
<Prilohy>
    <PredepsanaPriloha cislo="1"
                       nazev="Příloha účetní závěrky"
                       jm_souboru="04_Priloha_v_UZ_2025.pdf"
                       kod="PP_OPISPUV"
                       kodovani="base64">
<![CDATA[<base64-encoded PDF content>]]>
    </PredepsanaPriloha>
</Prilohy>
```

| Attribute     | Purpose                                                                       |
| ------------- | ----------------------------------------------------------------------------- |
| `cislo`       | Sequence number, starts at 1                                                  |
| `nazev`       | Display name shown in EPO ("Příloha účetní závěrky")                          |
| `jm_souboru`  | Original filename — preserve user-meaningful name                             |
| `kod`         | Příloha type code: `PP_OPISPUV` for "Opis původní" (standard příloha k 11. p.)|
| `kodovani`    | Always `base64`                                                               |

Generate the base64 payload via `base64 -i <file>.pdf -o /tmp/payload.txt`, then inject into the CDATA. Validate the resulting XML with `xmllint --noout`.

### What needs an embedded PDF

| Document                        | Embed PDF?                            |
| ------------------------------- | ------------------------------------- |
| Příloha v účetní závěrce        | yes (free-form prose)                 |
| Rozvaha                         | no (auto-rendered from VetaUA/UD)     |
| Výkaz zisku a ztráty            | no (auto-rendered from VetaUB)        |
| Smlouvy / notářské zápisy       | yes if applicable                     |
| Auditor's report                | yes if audit obligation triggered     |

For typical mikro `s.r.o.` filing only `04_Priloha_v_UZ_<year>.pdf` is embedded as Příloha č. 1.

## `<Kontrola>` element

The `<Kontrola><Soubor Delka="..." KC="..." Nazev="..." c_ufo="..." /></Kontrola>` footer that appears in EPO-saved files is **auto-generated by the EPO software** — never write it by hand.

| Attribute  | Meaning                              |
| ---------- | ------------------------------------ |
| `Delka`    | Total payload byte length            |
| `KC`       | Kontrolní součet (MD5 of payload)    |
| `Nazev`    | Filename without extension           |
| `c_ufo`    | Repeats the kraj kód from VetaD      |

When generating XML for upload, omit `<Kontrola>` entirely. EPO computes and inserts it on save/sign. If you receive an XML from an accountant that has `<Kontrola>` and you need to modify the payload, regenerate via EPO or recompute the checksum.
